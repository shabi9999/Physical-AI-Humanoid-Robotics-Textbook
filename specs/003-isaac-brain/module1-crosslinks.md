## Chapter 1: Isaac Sim Fundamentals → Module 1

### Core Connections

**From Module 1, Chapter 3 (URDF & Humanoid Structure)**:
- URDF files define robot structure that gets loaded into Isaac Sim
- Link/joint definitions determine physics simulation behavior
- Inertia parameters (mass, moments) are critical for realistic simulation
- Collision geometries defined in URDF become collision shapes in Isaac Sim

**Cross-link reference**:
```
See [Module 1, Chapter 3: URDF Model Description](/docs/module1/chapter3-urdf-model)

```

**From Module 1, Chapter 1 (ROS 2 Middleware)**:
- ROS 2 topics receive sensor data from Isaac Sim simulation
- Topic naming conventions (/camera/color/image_raw, /lidar/points, etc.) match Isaac Sim output
- QoS policies affect real-time sensor data streaming from simulation
- Service calls can control simulator pause/play, reset, frame stepping

**Cross-link reference**:
```
See [Module 1, Chapter 1: ROS 2 Communication](/docs/module1/chapter1-ros2-core)
to understand topic structures receiving Isaac Sim sensor data.
```

**From Module 1, Chapter 2 (Autonomous Agents)**:
- Agents running in simulation read sensor topics published by Isaac Sim
- Control commands from agents trigger motor movements in physics simulation
- Simulation cycles: sensors → perception → decision → action → physics → output
- Testing agents in simulation before real hardware deployment

**Cross-link reference**:
```
See [Module 1, Chapter 2: Autonomous Agents](/docs/module1/chapter2-agent-bridge)

## Chapter 3: Visual SLAM (Isaac ROS VSLAM) → Module 1

### Core Connections

**From Module 1, Chapter 1 (ROS 2 Nodes & Topics)**:
- Isaac ROS VSLAM runs as ROS 2 nodes
- Subscribes to /camera topics (RGB/D images from Isaac Sim or real cameras)
- Publishes /tf (transform frame) topics for robot pose
- Publishes /map topic for built environment maps
- Subscribes to /imu for inertial measurement correction

**Cross-link reference**:
```
See [Module 1, Chapter 1: ROS 2 Nodes](/docs/module1/chapter1-ros2-core#ros-2-nodes)
to understand node architecture of Isaac ROS VSLAM.
```

**From Module 1, Chapter 3 (Coordinate Frames & URDF)**:
- URDF defines /base_link, /camera_link coordinate frames
- Isaac ROS VSLAM publishes odom→base_link and map→odom transforms
- Frame relationships critical for robot localization
- TF (transform) tree management in ROS 2

**Cross-link reference**:
```
See [Module 1, Chapter 3: Coordinate Frames in URDF](/docs/module1/chapter3-urdf-model#coordinate-frames)
for understanding frame hierarchies in robot structure.
```

**From Module 1, Chapter 2 (Agent Decision-Making)**:
- Agents use /tf transforms to determine robot location
- Localization enables goal-based navigation
- Agents subscribe to /map topic for environment awareness
- Decision logic: "Where am I?" → "Where should I go?" → Plan path

**Cross-link reference**:
```
See [Module 1, Chapter 2: Agent Perception](/docs/module1/chapter2-agent-bridge#agent-perception)
to understand how agents use localization data for decision-making.
```

## Chapter 4: Navigation 2 (Nav2) Path Planning → Module 1

### Core Connections

**From Module 1, Chapter 1 (ROS 2 Actions & Services)**:
- Nav2 exposes ROS 2 Action servers (nav2_msgs/NavigateToPose)
- Agents call Nav2 actions: "Navigate to (x, y, theta)"
- Feedback mechanism: Progress updates during path execution
- Result mechanism: Success or failure of navigation attempt

**Cross-link reference**:
```
See [Module 1, Chapter 1: ROS 2 Actions](/docs/module1/chapter1-ros2-core#ros-2-actions)
to understand action-based navigation interfaces.
```

**From Module 1, Chapter 3 (Footprint & Collision)**:
- URDF defines robot footprint (size, shape for collision checking)
- Nav2 costmap uses footprint to avoid collisions
- Collision geometry from URDF → Nav2 collision checking
- Safe navigation depends on accurate robot geometry

**Cross-link reference**:
```
See [Module 1, Chapter 3: Robot Geometry](/docs/module1/chapter3-urdf-model#collision-geometry)
for understanding how robot size affects navigation safety.
```

**From Module 1, Chapter 2 (Agent-based Navigation)**:
- Agents call Nav2 to reach goals
- Navigation feedback updates agent world model
- Agents respond to navigation failures (replan, retry, abort)
- Integration: Agent → Nav2 action server → robot motion

**Cross-link reference**:
```
See [Module 1, Chapter 2: Goal-Based Agents](/docs/module1/chapter2-agent-bridge#goal-based-agents)
for understanding how agents command navigation.
